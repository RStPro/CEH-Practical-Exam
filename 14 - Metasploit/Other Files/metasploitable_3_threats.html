Metasploitable3 — Intentionally Malicious Files & Vulnerable Services

This document lists the common backdoors, web shells, vulnerable apps, misconfigurations, and safety advice you can expect to find in Metasploitable3. The VM is intentionally insecure for learning and testing; treat it like a live hostile environment and keep it isolated.

1. Web Shells & Backdoors

Antivirus products may flag these as Backdoor, Webshell, or similar.

  • Backdoor:ASP/Dirtelti — ASP-based webshells or malicious ASP scripts that enable remote command execution.
  • PHP webshells (e.g., c99.php, r57.php) — common in older PHP apps and intentionally included in web directories.
  • JSP web shells — present in older Java/Tomcat web apps.

2. Vulnerable Web Applications

These apps contain known vulnerabilities and often include exploit-ready files:

  • DVWA (Damn Vulnerable Web App)
  • Mutillidae II
  • WebGoat
  • Joomla (outdated versions) with RCE/SQLi vulnerabilities
  • Drupal (old versions) — e.g., "Drupalgeddon" style issues
  • ColdFusion 8 — outdated admin panels and endpoints
  • Tomcat Manager with default or weak credentials

3. Misconfigured & Weak Services

Services intentionally left insecure to practice exploitation and post-exploitation:

  • FTP with anonymous login and writable directories
  • Telnet (cleartext login)
  • SMB shares accessible without proper permissions
  • VNC with no or weak authentication
  • MySQL root with empty password
  • PostgreSQL with default credentials
  • Various unpatched services and old packages (Windows & Linux)

4. Intentional Malware Samples & Payloads

  • Stored Metasploit payloads and executables in web directories for detection and practice.
  • Reverse shell scripts (*.sh, *.bat, *.asp, *.php).
  • Meterpreter binaries used in demos and labs.

Safety Precautions

  • Network isolation: Always run Metasploitable3 on a Host-Only or Internal network. Do not expose it to your home/office network or the internet.
  • No shared folders: Avoid mapping host folders into the VM to prevent accidental contamination of your host filesystem.
  • Snapshots: Take a clean snapshot after initial deploy so you can revert when needed.
  • Use only in lab: Never use these images for production purposes.
  • Data handling: Do not upload suspicious files from the VM to public services or email them.
